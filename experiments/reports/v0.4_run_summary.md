## v0.4 Conclusion

True-gated BitDPM validates parameter-block controllability and reveals block interference. A fixed beneficial block configuration, `Only chinese blocks`, improves over baseline, while all-block, random, keyword, and entropy routing degrade to 0.778. This indicates that BitDPM's main bottleneck is not block execution overhead or parameter availability, but accurate block utility routing.

## v0.5 Direction

BitDPM v0.5 should focus on router calibration and block utility learning:

- Oracle per-sample router: estimate the upper bound if the router always selects the best available block configuration.
- Block utility matrix: identify which prompts/categories benefit from each block type.
- Validation-calibrated router: choose blocks from measured validation utility instead of hand-written keyword or entropy rules.

NF4 is deferred. The NF4 backend remains compatible in design, but full-model conversion is currently bottlenecked by device memory and CPU conversion time. v0.4 therefore focuses on corrected gating and block-routing behavior; NF4 recovery should wait for an optimized backend.
