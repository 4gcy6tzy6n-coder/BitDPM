# BitDPM v0.6: Block Utility Amplification

**Date**: 2026-06-07

## Status

BitDPM v0.6 block amplification run is complete.

Previous v0.5 validated state:

- Controllable parameter-block activation works.
- All-block composition introduces interference or no gain.
- Current blocks do not provide stable deterministic utility.
- Sampling generation shows sparse block utility: oracle improves from 0.800 to 0.844, but selects non-baseline blocks for only 2/45 samples.

v0.6 update:

- Expanded-data rank-8 blocks trained successfully.
- Deterministic fixed block utility appeared: `code` and `chinese` reached 0.800 vs baseline 0.778.
- Deterministic oracle improved to 0.822, selecting non-baseline blocks for 2/45 samples.
- Sampling fixed block utility improved for `code`: 0.822 vs baseline 0.800.
- Sampling oracle remained 0.844 with only 2/45 non-baseline selections.

Main bottleneck after v0.6:

> Block quality improved enough to create fixed-block gains, but useful block utility remains sparse. The next leverage point is not router complexity; it is increasing the number of samples where blocks beat baseline.

## v0.6 Objective

Amplify block utility so that blocks create stable, measurable task gains.

Success criteria:

1. At least one fixed block type beats baseline under deterministic generation.
2. Sampling oracle non-baseline selections increase from 2/45 to at least 8/45.
3. Always-All remains lower than selective/oracle activation, preserving the interference finding.

v0.6 outcome:

| Criterion | Result | Status |
|---|---|---|
| Deterministic fixed block > baseline | `code=0.800`, `chinese=0.800` vs `baseline=0.778` | Passed |
| Sampling oracle non-baseline selections >= 8/45 | `2/45` | Failed |
| Always-All lower than selective/oracle | deterministic: `always_all=0.778 < oracle=0.822`; sampling: `always_all=0.800 < oracle=0.844` | Passed |

## Experiment A: More Training Data

Hypothesis:

Current block training data is too small and too weakly differentiated.

Plan:

- Expand each category to 500-2000 training prompts.
- Keep categories: `general`, `math`, `code`, `chinese`.
- Train blocks with the same target modules first: `o_proj`, `down_proj`.
- Use the same v0.5 matched-setting protocol for evaluation.

Interpretation:

v0.6 confirms that stronger block training can produce deterministic fixed-block gains. This directly improves over v0.5, where deterministic oracle selected baseline for all samples. However, oracle utility remains sparse: only 2/45 samples benefit from non-baseline activation in both deterministic and sampling settings.

The best current statement is:

> BitDPM v0.6 achieves the first stable deterministic block utility signal: fixed `code` and `chinese` blocks outperform the deterministic baseline. However, useful per-sample block utility remains sparse, so routing headroom is still limited. The next stage should increase utility coverage rather than introduce more router variants.

## Experiment C: Task-Utility Training

Hypothesis:

Next-token prompt reconstruction is not aligned enough with benchmark utility.

Plan:

- Build answer-format training examples for each category.
- For math/common QA, force concise answer-first outputs.
- Add hard negatives where current baseline misses expected keyword.
- Train blocks to push correct answer tokens into greedy top-1, not just shift style.

Evaluation:

Use deterministic v0.5 first. Sampling gains are secondary until deterministic utility appears.

## Deprioritized Work

Do not spend more time on these until block utility is stronger:

- NF4 full-model recovery
- Additional EntropyRouter variants
- More active block budget scans
- Complex router learning

## Current Project Statement

BitDPM v0.6 block amplification completed. The controllable parameter-block framework is validated, and stronger training produced the first stable deterministic fixed-block gains: `code` and `chinese` reached 0.800 vs deterministic baseline 0.778. Sampling fixed-block utility also improved for `code` at 0.822 vs baseline 0.800. However, oracle non-baseline selection remains sparse at 2/45 in both deterministic and sampling settings. The next stage should focus on increasing utility coverage through task-utility training, harder missed-example data, and possibly capacity scans, not router complexity.
