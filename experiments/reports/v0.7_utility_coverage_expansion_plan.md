# BitDPM v0.7: Utility Coverage Expansion

## Status

Code scaffold completed. The user will run the experiments.

## Objective

v0.6 proved that stronger block training can produce deterministic fixed-block gain:

- deterministic baseline: 0.778
- deterministic code/chinese: 0.800
- sampling code: 0.822
- sampling oracle: 0.844
- oracle non-baseline coverage: 2/45

v0.7 targets the remaining bottleneck: increase useful block coverage from 2/45 to a visibly larger fraction.

## Success Criteria

- Oracle non-baseline selection increases beyond 2/45, ideally 8/45 or higher.
- At least one deterministic fixed-block score remains above 0.800.
- Sampling oracle improves beyond the v0.6 0.844 level.
- Always-All remains no better than the best selective/fixed configuration.

## Code Added

- `bitdpm/train/v07_data.py`: curated answer-bearing v0.7 datasets.
- `scripts/train_v07_blocks.py`: trains hard-sample, error-type, and structure blocks.
- `scripts/run_v05_router_calibration.py`: now supports dynamic block types, `--configs`, and `--tag`.
- `bitdpm/train/train_blocks.py`: `train_block()` now accepts explicit training texts.

## Reading the Result

The primary v0.7 metric is `selection_frequency`: non-baseline oracle choices should increase beyond 2/45. If fixed-block deterministic scores increase but oracle coverage remains 2/45, block strength improved but coverage did not. If oracle coverage increases while fixed-block average stays flat, the blocks are useful but sparse and need routing only after coverage becomes meaningful.
