# BitDPM v0.7 Utility Coverage Round Summary

Date: 2026-06-07

## Goal

v0.7 的目标是从 v0.6 的状态继续推进：

- v0.6 已证明增强 block 训练能产生 deterministic fixed-block gain。
- v0.6 sampling oracle 为 `0.844`，但 non-baseline oracle coverage 只有 `2/45`。
- v0.7 重点不是 router，而是扩大 block utility coverage。

核心问题：

> 能否通过 hard-sample 数据、error-type 数据、连续层结构和 scale calibration，让更多样本存在可用 block utility？

## Important Evaluation Note

普通 sampling 结果会受 config 数量和顺序影响，因为每次 generation 会消耗 RNG。

因此：

- 普通 sampling 结果适合看趋势。
- 需要比较 full configs 和 selected configs 时，必须使用 `--stable-sampling-seeds`。
- stable sampling 下每个 `(sample, config)` 使用固定 seed，结果更可比。

## Experiment Matrix

### 1. Hard-Sample Blocks, `l23_down`

训练数据：

- `hard_math`
- `hard_commonsense`

结果：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | baseline 0.778 | 0.778 | 0/45 | 0.778 |
| sampling | 0.778 | hard_commonsense 0.800 | 0.800 | 1/45 | 0.800 |

结论：

- hard-sample targeted data 没有打开 coverage。
- sampling 下只帮助 1 个样本：`Who wrote Romeo and Juliet?`

### 2. Error-Type Blocks, `l23_down`

训练数据：

- `calculation_error`
- `commonsense_choice`
- `format_following`
- `chinese_semantic`
- `short_reasoning`

结果：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | format_following 0.800 | 0.800 | 1/45 | 0.778 |
| sampling | 0.800 | baseline 0.800 | 0.822 | 1/45 | 0.789 |

结论：

- error-type block 比 hard-sample 更有价值。
- `format_following` 在 deterministic 下产生 fixed-block gain。
- 但 coverage 仍只有 `1/45`。

### 3. Error-Type Blocks, `l22_l24_down`

每个 error-type 激活 3 个 down_proj blocks。

默认 `block_scale=0.15` 时：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | baseline 0.778 | 0.778 | 0/45 | 0.778 |
| sampling | 0.800 | chinese_semantic 0.822 | 0.844 | 2/45 | 0.800 |

结论：

- 连续 down_proj 结构在 sampling 下恢复到 v0.6 oracle 水平。
- 但 deterministic 增益消失。
- 初步怀疑多 block 平均分摊 scale 后，每个 block effect 太弱。

### 4. Error-Type Blocks, `l23_ffn`

每个 error-type 激活 `up_proj + down_proj`。

默认 `block_scale=0.15` 时：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | baseline 0.778 | 0.778 | 0/45 | 0.778 |
| sampling | 0.800 | short_reasoning 0.822 | 0.822 | 1/45 | 0.800 |

`block_scale=0.30` 时：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | baseline 0.778 | 0.778 | 0/45 | 0.778 |
| sampling | 0.778 | commonsense_choice 0.822 | 0.822 | 2/45 | 0.800 |

结论：

- `l23_ffn` 没有保住 deterministic gain。
- scale 提高后 sampling coverage 到 `2/45`，但 oracle 不超过 `0.822`。
- 不如 `l22_l24_down`。

## Scale Calibration on `l22_l24_down`

这是本轮最关键部分。

### Scale 0.45

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | chinese_semantic / commonsense_choice / format_following 0.800 | 0.822 | 2/45 | 0.778 |
| sampling | 0.778 | commonsense_choice / format_following 0.822 | 0.844 | 3/45 | 0.778 |

结论：

- scale calibration 证实多 block 结构不是无效，而是默认 scale 太弱。
- deterministic fixed-block gain 恢复。
- sampling oracle 到 `0.844`。

### Scale 0.60

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | commonsense_choice 0.822 | 0.822 | 2/45 | 0.778 |
| sampling | 0.800 | commonsense_choice 0.822 | 0.844 | 2/45 | 0.800 |

结论：

- deterministic best fixed 到 `0.822`。
- 但 sampling coverage 没继续扩大。

### Scale 0.75

普通 sampling：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| deterministic | 0.778 | commonsense_choice 0.822 | 0.822 | 2/45 | 0.789 |
| sampling | 0.800 | commonsense_choice 0.844 | 0.867 | 3/45 | 0.800 |

stable sampling：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| stable sampling | 0.822 | baseline / commonsense_choice 0.822 | 0.867 | 2/45 | 0.778 |

结论：

- `block_scale=0.75` 是本轮最强 sampling oracle：`0.867`。
- stable sampling 下 oracle `0.867` 仍成立，说明不是 RNG 假象。
- 但 stable coverage 是 `2/45`，不是普通 sampling 的 `3/45`。
- 高 scale 开始损伤部分 block：
  - `calculation_error`
  - `format_following`
  - `short_reasoning`
- 不能继续盲目升 scale。

## Selected-Block Evaluation

目的：

只保留 aggregate 上较好的 block：

- `chinese_semantic`
- `commonsense_choice`
- `format_following`

去掉：

- `calculation_error`
- `short_reasoning`

结果：

| Setting | Baseline | Best Fixed | Oracle | Non-Baseline Oracle | Always-All |
|---|---:|---:|---:|---:|---:|
| selected deterministic | 0.778 | commonsense_choice 0.822 | 0.822 | 2/45 | 0.789 |
| selected stable sampling | 0.822 | baseline / commonsense_choice 0.822 | 0.844 | 1/45 | 0.778 |

结论：

- selected 没有提升，反而降低 oracle。
- `calculation_error` 虽然 aggregate 很差，但在 `What is 2^10?` 上是唯一成功 block。
- 因此不能按 aggregate score 粗暴删除 block。

关键现象：

> BitDPM block utility 是低平均、高稀疏的。一个整体表现差的 block 仍可能对某个样本有唯一 utility。

## Main Findings

### 1. Hard-Sample Training Alone Was Not Enough

Hard-sample targeted blocks produced at most sampling `0.800` and coverage `1/45`.

This suggests that simply training on benchmark-like questions is not sufficient to make blocks broadly useful.

### 2. Error-Type Blocks Are Better Than Task-Type or Hard-Sample Blocks

`format_following`, `commonsense_choice`, and `chinese_semantic` repeatedly produced fixed-block gains.

This supports the v0.7 hypothesis:

> Blocks are more naturally error-repair modules than domain experts.

### 3. Structure Matters

`l22_l24_down` with calibrated scale was the best structure.

`l23_ffn` did not improve deterministic behavior and did not beat `l22_l24_down` in sampling oracle.

### 4. Scale Calibration Is Essential

Default `block_scale=0.15` under-powered multi-block configs.

For `l22_l24_down`:

- 0.15: weak deterministic effect.
- 0.45: deterministic gain returns.
- 0.60: best deterministic fixed score reaches `0.822`.
- 0.75: best sampling oracle reaches `0.867`, but some blocks degrade strongly.

### 5. Always-All Still Shows Interference

Across most runs, Always-All is no better than baseline and often worse.

At high scale, Always-All remains below oracle/selective behavior:

- stable sampling scale 0.75:
  - baseline `0.822`
  - oracle `0.867`
  - Always-All `0.778`

This continues to support the core interference claim.

### 6. Aggregate Filtering Is Unsafe

Selected-block eval showed that removing low-average blocks can remove unique oracle utility.

`calculation_error` is the clearest case:

- poor aggregate score
- uniquely solves `What is 2^10?`

This means future router/block selection should estimate per-sample utility, not average block quality alone.

## Updated v0.7 Conclusion

BitDPM v0.7 improves over v0.6 by showing that block utility can be amplified through error-type training, contiguous down-proj placement, and scale calibration. The best stable sampling result reaches oracle `0.867`, exceeding the v0.6 sampling oracle of `0.844`, while deterministic best fixed-block performance improves to `0.822`. However, utility coverage remains sparse: the best stable sampling run selects non-baseline blocks for only `2/45` samples. The main bottleneck is therefore still utility coverage, not router sophistication.

中文总结：

v0.7 证明了 error-type 训练、连续 `l22-l24 down_proj` 结构和 scale calibration 可以继续放大 BitDPM block utility。最佳 stable sampling oracle 从 v0.6 的 `0.844` 提升到 `0.867`，deterministic best fixed-block 也提升到 `0.822`。但 block utility 仍然高度稀疏，最佳 stable sampling 下只有 `2/45` 个样本选择非 baseline。当前瓶颈仍是 utility coverage，而不是 router。

## Suggested Next Direction

下一阶段不建议继续做：

- selected aggregate filtering
- 更复杂 router
- 盲目增加 scale

更值得做的是：

1. 扩大 evaluation benchmark，确认 `2/45` 是否只是小 benchmark 下的稀疏表现。
2. 针对 oracle-selected 和 near-miss samples 构造更多同型训练数据。
3. 做 per-sample utility mining，而不是按 block aggregate 删除或保留。
4. 针对 `l22_l24_down` 做更系统的 capacity scan，例如 rank 16 / rank 32，而不是继续升 scale。
